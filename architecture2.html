How Multi-Hop Preserves E2E (Nested Encryption)
Scenario A — Yggdrasil over WireGuard (recommended path) Client │ X25519 encrypt to dest Yggdrasil pubkey ← E2E layer — only dest can read ▼ Yggdrasil packet (ciphertext — intermediate nodes blind) │ wrapped in WireGuard ChaCha20-Poly1305 ← transport layer — VPN server can see Ygg packet but NOT its payload ▼ softether-vpn → lbrtynet peer mesh │ WireGuard terminates at peer's VPN server ▼ Peer's Yggdrasil router → hops to destination Ygg node │ destination holds the Ed25519 private key ▼ DECRYPTS ✓ E2E intact — WireGuard saw only ciphertext Scenario B — Skywire over Yggdrasil over WireGuard (double E2E) Client │ Noise_XK to dest Skywire pubkey ← E2E layer 1 — Skywire ▼ Skywire packet │ X25519 to dest Yggdrasil addr ← E2E layer 2 — Yggdrasil ▼ Yggdrasil packet │ ChaCha20-Poly1305 WireGuard ← transport — sees Ygg ciphertext only ▼ WireGuard → Yggdrasil → Skywire → destination node DECRYPTS both layers in reverse ✓ both E2E intact, nested Scenario C — Tor → clearnet (E2E BROKEN — for reference) Client │ AES-CTR × 3 onion layers ← hides your IP address ▼ Guard relay → Middle relay → Exit relay │ Exit DECRYPTS outer layer sends plaintext HTTP to clearnet server ← exit reads traffic Use HTTPS to clearnet if you must use Tor this way. Scenario D — I2P service tunnel (lbrtynet-comm / lbrtynet-httpd) Remote I2P peer │ ElGamal + AES garlic cloves to leaseset key ← E2E — only your node has the leaseset private key ▼ 6+ I2P routers (blind forwarding) ▼ i2pd on this node → local service on 127.0.0.1:5557 or :8080 DECRYPTS ✓ E2E intact — no IP of sender is known to the local service
